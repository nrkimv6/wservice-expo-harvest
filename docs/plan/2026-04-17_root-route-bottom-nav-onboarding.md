## 요구사항 정리

1. `/app` 대신 `/`를 실제 앱 진입점으로 올리고, `/`가 왜 존재하는지 모호한 상태를 정리한다.
2. 한 페이지에 길게 이어진 정보 배치를 하단 네비게이션 중심 구조로 바꿔 상황별 중요도를 분리한다.
3. `/app` 상단 소개 카드(`Map First`, `Search + Filter`, `Separate State`)는 상시 노출 대신 첫 실행 1회 모달로 이동한다.

## 현재 코드 관찰

- `src/routes/+page.svelte`: 마케팅 랜딩 성격의 소개 페이지이며, 실제 사용 흐름은 `/app` 링크로 넘긴다.
- `src/routes/app/+page.svelte`: 알림 배너, 소개 카드, 박람회 선택, 요약 카드, 지도, 리스트와 상태 로직이 한 파일에 모두 쌓여 있다.
- `src/lib/stores/farmState.ts`: 전시회별 로컬 상태와 마지막 선택 박람회만 저장하며, 온보딩 노출 여부는 저장하지 않는다.
- `src/service-worker.ts`: 앱 셸과 네비게이션 fallback이 `/app`을 포함한 현재 라우팅을 전제로 한다.
- `README.md`: 메인 동작 설명과 오프라인 진입 설명이 `/app` 기준으로 작성되어 있다.

## 구현 순서 제안

### Phase 1: 루트 라우트 정리

1. [ ] **실사용 엔트리를 `/`로 통합한다** — 랜딩과 앱 진입을 분리하지 않고 모바일 앱의 기본 URL을 하나로 고정
   - [ ] `src/routes/+page.svelte`: 현재 랜딩 마크업을 제거하고, 기존 `/app`의 상태·파생값·핸들러를 옮겨 실제 파밍 화면을 루트에서 바로 렌더링하도록 재구성한다.
   - [ ] `src/routes/+page.svelte`: `AlertBanner`, `ExhibitionMap`, `LootFeed`, `BoothDetailSheet` 연결과 메타데이터를 루트 진입 기준으로 다시 배치한다.

2. [ ] **`/app` 호환 진입을 가볍게 정리한다** — 기존 북마크와 PWA 진입을 깨지 않으면서 주 진입점만 `/`로 단순화
   - [ ] `src/routes/app/+page.svelte`: 기존 전체 UI를 제거하고 `/` 이동만 담당하는 가벼운 호환 라우트로 교체한다.
   - [ ] `src/service-worker.ts`: navigate fallback과 앱 셸 캐시를 `/` 우선으로 바꾸되, `/app`은 기존 링크 호환 경로로 유지할지 명시적으로 정리한다.

### Phase 2: 하단 네비게이션 정보 구조

3. [ ] **하단 네비게이션용 탭 상태를 설계한다** — 한 화면에 몰린 정보를 `홈/지도/리스트/저장됨` 성격으로 분리
   - [ ] `src/routes/+page.svelte`: 현재 선택 박람회와 선택 부스 상태를 유지한 채 탭 상태와 탭별 파생 데이터(`bookmarkedItems`, `completedItems`)를 추가한다.
   - [ ] `src/app.css`: 하단 고정 네비게이션, safe-area padding, 바텀시트와 겹치지 않는 기본 여백 유틸을 추가한다.

4. [ ] **홈 탭을 요약 중심으로 재배치한다** — 항상 봐야 하는 정보만 기본 탭에 남기고 초기 밀도를 낮춘다
   - [ ] `src/routes/+page.svelte`: `AlertBanner`, 박람회 선택, 핵심 카운트, 도움말 재오픈 버튼을 홈 탭에 묶어 상단 우선 정보로 정리한다.
   - [ ] `src/routes/+page.svelte`: 현재 긴 단일 스크롤에서만 쓰이던 구분선과 상시 소개 카드 영역을 제거해 첫 화면을 짧게 만든다.

5. [ ] **지도·리스트·저장됨 탭을 각각 연결한다** — 탭을 바꿔도 현재 박람회 맥락과 상세 시트가 끊기지 않게 유지
   - [ ] `src/routes/+page.svelte`: 지도 탭에는 `ExhibitionMap`, 리스트 탭에는 전체 `LootFeed`, 저장됨 탭에는 찜/완료 기준 subset 데이터를 각각 연결한다.
   - [ ] `src/routes/+page.svelte`: 탭 전환 중에도 선택 부스 시트, 찜/완료 토글, 메모 저장이 동일한 상태 소스를 쓰도록 유지한다.

6. [ ] **하위 컴포넌트를 탭 문맥에 맞게 다듬는다** — 기존 컴포넌트 문구와 빈 상태가 단일 페이지 전제를 덜 갖도록 조정
   - [ ] `src/lib/components/LootFeed.svelte`: 리스트 제목, 보조 설명, 빈 상태 문구를 prop 기반으로 바꿔 기본 리스트와 저장됨 리스트를 모두 자연스럽게 렌더링한다.
   - [ ] `src/lib/components/ExhibitionMap.svelte`: 지도 단독 탭에서 헤더 설명과 카드 밀도가 과하지 않도록 문구와 간격을 정리한다.

### Phase 3: 첫 실행 온보딩 모달

7. [ ] **소개 카드를 첫 실행 1회 모달로 옮긴다** — 상시 노출 대신 처음 진입할 때만 핵심 사용법을 안내
   - [ ] `src/routes/+page.svelte`: 기존 `Map First`, `Search + Filter`, `Separate State` 카드를 첫 실행 모달 콘텐츠로 옮기고, 홈 탭에서만 재오픈 가능한 도움말 버튼과 연결한다.
   - [ ] `src/app.css`: 모달 오버레이, 백드롭, 스크롤 잠금, 바텀 네비 z-index 충돌을 제어하는 스타일을 보강한다.

8. [ ] **온보딩 노출 여부를 로컬에 저장한다** — 이후 재방문에서는 자동으로 다시 뜨지 않게 처리
   - [ ] `src/lib/stores/farmState.ts`: 온보딩 완료 여부를 읽고 저장하는 localStorage helper를 추가한다.
   - [ ] `src/routes/+page.svelte`: 최초 1회 자동 오픈, dismiss 저장, “다시 보기” 수동 재진입 흐름을 같은 상태 모델로 연결한다.

### Phase 4: 회귀 점검 및 문구 정리

9. [ ] **새 구조 기준으로 카피와 메타 정보를 정리한다** — 루트 진입과 탭 구조가 문서/메타에 반영되도록 정리
   - [ ] `src/routes/+page.svelte`: 페이지 title/description과 홈/지도/리스트/저장됨 탭 카피를 새 엔트리 구조에 맞게 조정한다.
   - [ ] `README.md`: 현재 포함 범위 설명에서 랜딩 표현을 줄이고, 루트 엔트리·하단 네비·첫 실행 모달 구조를 반영한다.

10. [ ] **정적 검증으로 라우트/타입 회귀를 확인한다** — 문서와 코드 변경이 SvelteKit 빌드 기준과 충돌하지 않는지 검증
   - [ ] `package.json`: 기존 `check` 스크립트를 기준으로 `npm run check`를 실행해 라우트/타입 오류가 없는지 확인한다.
   - [ ] `src/service-worker.ts`: `/app` 잔존 참조가 호환 목적 외에 남지 않도록 최종 점검하고, 불필요한 fallback 분기를 제거한다.

## 기대 결과

- 사용자는 `/`에서 바로 앱을 시작하고, `/app`은 과거 링크 호환용 진입점으로만 남는다.
- 첫 화면은 길게 늘어진 종합 페이지가 아니라, 하단 네비게이션으로 목적별 화면을 빠르게 오갈 수 있는 구조가 된다.
- 소개 카드는 첫 진입에만 보여 주고, 이후에는 화면을 가리지 않으면서 필요 시 다시 열 수 있다.

## 리스크 및 확인 포인트

- `/app`을 바로 제거하면 기존 북마크나 설치형 PWA 진입이 깨질 수 있으므로 최소한 한 차례는 리다이렉트 호환이 필요하다.
- 하단 네비게이션을 넣으면 기존 상세 바텀시트와 safe-area 여백이 겹칠 수 있어 모바일 실기기 간격 점검이 필요하다.
- 첫 실행 모달 상태 키를 추가할 때 기존 로컬 저장 스키마와 충돌하지 않도록 별도 키로 분리해야 한다.
